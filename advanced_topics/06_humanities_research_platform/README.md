# Advanced Lesson 6 — Capstone: Building a Humanities Research Platform

## Goal

Assemble the advanced course into a small humanities research platform.

The platform is not just a chat transcript.

It is a reproducible research environment that combines:

```text
research policy
corpus
theory library
skill
RAG context packet
loop orchestration
research harness
annotation assets
model adaptation memo
final report
```

---

## Course Synthesis

The advanced course has built the platform piece by piece.

```text
Lesson 1:
Package a humanities method as a skill and understand tools/MCP.

Lesson 2:
Ground the method in retrieved context through RAG.

Lesson 3:
Run the method through loops and role orchestration.

Lesson 4:
Wrap the method in a research harness: policy, resources, checks, evaluation, archive.

Lesson 5:
Turn outputs into annotation assets and decide whether model adaptation is justified.

Lesson 6:
Assemble everything into a humanities research platform.
```

---

## Platform Architecture

A platform has a repeatable structure:

```text
Research question
→ corpus and theory library
→ skill
→ RAG/context engineering
→ loop/orchestration
→ research harness
→ annotation assets
→ model adaptation decision
→ final interpretive report
```

Another view:

```text
Corpus
→ Theory Library
→ Skills
→ RAG
→ Agents / Loops
→ Research Harness
→ Annotation Assets
→ Reports
```

## What Makes This a Platform?

A platform is more than an answer.

It has:

```text
documented inputs
defined methods
allowed sources
reusable prompts
retrieval strategy
loop rules
evaluation criteria
external checks
saved outputs
annotation assets
final reflection
```

A platform should let another student or researcher understand:

```text
what was asked
what sources were used
what method was applied
what the model was allowed to do
what was checked
what failed
what was revised
what remains uncertain
```

## Capstone Workflow

### Step 1 — Define the research question

Example:

```text
How do metaphors of fragmentation and reflection express loss in short urban lyric passages?
```

### Step 2 — Define the corpus

List the source texts.

Include:

- title or text ID
- source
- date if known
- genre
- notes
- permissions or copyright status if relevant

### Step 3 — Define the theory library

Include theory notes or concepts.

Example:

```text
metaphor theory
source / target domain
affect
ambiguity
negative / positive liberty
narrative transition
```

### Step 4 — Define the skill

Use or adapt a skill such as:

```text
metaphor_analysis_skill
translation_comparison_skill
discourse_analysis_skill
narrative_structure_skill
```

### Step 5 — Build RAG context

Create at least one context packet that includes:

```text
primary text
theory note
prior annotation or taxonomy
research question
instructions
```

### Step 6 — Run a loop

Run at least one loop:

```text
Analyst → Skeptic → Editor → Evaluator → Archivist
```

Save each stage.

### Step 7 — Wrap the method in a research harness

Define:

```text
allowed sources
forbidden claims
citation policy
uncertainty policy
tools/checks
rubric
human review points
archive rules
```

### Step 8 — Create annotation assets

Create:

```text
annotation_schema.json
annotations.jsonl
few_shot_prompt.md
evaluation_split.jsonl
```

### Step 9 — Write model adaptation memo

Decide:

```text
Should this remain a skill/RAG/harness method?
Is fine-tuning justified?
Would distillation or LoRA make sense later?
What data is missing?
```

### Step 10 — Write final report

The final report should include:

- research question
- method
- findings
- evidence
- uncertainty
- evaluation results
- limitations
- next steps

## Student Success Criteria

A successful project does not need to build a production app.

A successful project shows:

```text
methodological clarity
source grounding
reproducibility
explicit uncertainty
evidence-based interpretation
appropriate use of tools
thoughtful evaluation
human review
honest limitation statements
```

The final interpretation does not need to be definitive.

It must be accountable.

---

## Common Failure Modes

Watch for:

```text
beautiful final answer but no method
prompts without saved outputs
RAG packet without source boundaries
loop without stopping condition
harness without actual checks
annotations without review status
fine-tuning memo that assumes training is automatically better
unclear distinction between evidence and interpretation
no human review points
```

## Final Reflection Questions

1. What did the platform make repeatable?
2. What did RAG contribute?
3. What did the loop improve?
4. What did the harness constrain or check?
5. What did the annotations preserve?
6. Why is or is not fine-tuning justified?
7. What remains human judgment?
8. What would be needed to use this platform on a new corpus?
